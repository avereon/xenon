Getting Started

Welcome, and thank you for using Xenon. Xenon is a simple application framework that provides services common to modern desktop applications like selection tabs, dockable work areas, notifications and many others. We hope you find the framework useful as you get to know its features and capabilities. We particularly encourage you to discover the modules available for Xenon that provide capabilities to accomplish all kinds of tasks.

Download and Install

If you are new to Xenon, the installers for current versions are found on the Xenon product page. Once downloaded, run the installer to install Xenon. When installed Xenon can be found by searching for Xenon in the start menu or application finder of your operating system.

Terminology

In the course of reading this document, you will run across a number of terms that have specific meaning in Xenon. This section serves as a brief introduction to some of the concepts and terminology used in Xenon as well as an explanation for each.

Program
Xenon uses the term 'program' to refer to the program itself. This term is used in place of 'application' for simplicity and to avoid ambiguity with the term application or app.
Resource
A resource is something on which Xenon operates. Specifically resources are things that are 'opened' and 'closed' and sometimes 'loaded' and 'saved'. Resources are defined by Uniform Resource Identifiers (URIs) and a URI is used to identify a specific resource to Xenon. Some examples of resources are:
  • An SSH connection to another computer.
  • A file on the local file system.
  • The built-in about resource.
Resource Type
Resources are always assigned a resource type in order to identify what tools are available for that resource. If a resource type cannot be determined then Xenon is unable to operate on that resource. Resource types are also used to determine what tools can be used on a particular resource. Resource types are very similar to media types as defined by IANA (https://www.iana.org/assignments/media-types/media-types.xhtml) and are commonly associated directly with a media type.
Workspace
A workspace is everything in a Xenon window. Xenon allows multiple workspaces to be open at any given time. Workspaces allow the user to switch between workareas quickly.
Workarea
A workarea is a named collection and layout of tools.
Tool
A tool is a graphical component that provides specific functionality to operate on a resource. Some examples of tools are: a text editor, an image viewer, a calculator and an SSH terminal. Tools are provided and implemented by modules.
Action
Actions define the set of operations that can be used with tools. Many actions are common among tools (e.d. cut, copy, paste) and some actions are specific. Tools use common actions to provide a simpler user experience.
Program Data Folder
This is the location where the program configuration and other files that are not part of the program itself are located. The location of the program home folder is different across platforms but the structure is the same. The following table shows the default program data folder location for common platforms:
Linux $HOME/.config/xenon
Mac OS $HOME/Library/Application Support/Xenon
Windows %HOME%\AppData\Roaming\Xenon
Program Home Folder
This is the location where the program is installed and from where the program executes. The location of the program home is different across platforms but the structure is the same. The following table shows the default program home location for common platforms:
Linux $HOME/.local/share/xenon
Mac OS /Applications/Xenon.app/Contents
Windows C:\Program Files\Xenon
Product
The program and modules are collectively referred to as products. You will notice that program updates and modules updates are handled with the same tools and workflow. Anytime you see the term product it may refer to either the program, modules or both.
Modules
A modules is a binary package that provides specific functionality. In specific, it provides tools to operate on resources. Mods may also provide additional resource types and actions that can be used by other modules.
Product Market
A network location, usually on the Internet, where to find available products.

Command Line Interface

Xenon provides a command line interface to start and interact with the program. The command line interface requires that the program be runnable from the command line. It is expected that users use the provided program shortcuts and shell scripts such that Xenon may simply be started by using the command 'xenon' on the command line from any location. This document assumes that Xenon is available to execute in this manner:

xenon [<option>...] [<uri>...]

Advanced: Xenon is a Java module and as such can be started using the Java module command line parameters. Xenon has been distributed as a jlink-ed program and should be executed using the provided Java runtime. To start Xenon using the Java module commands use the following:

${XENON_HOME}/bin/java -m com.avereon.xenon

URIs (Resources)

Providing a URI to Xenon on the command line instructs Xenon to treat the URI as a resource and attempt to open that resource after starting. Any valid URI should be acceptable but may not be able to be supported by Xenon if a supporting module has not been installed. Any URI that does not specify a scheme will be interpreted as a file URI. Relative URIs will be resolved against the user.path system property of the invoking JVM.

Basic Options

--help
Print the command line help information. This option exits the invoked instance immediately upon completion. It does not affect already running instances.
--version
Print the program version information. This option exits the invoked instance immediately upon completion. It does not affect already running instances.

Control Options

Several options control the operation of the program allowing some level of automation. If Xenon is run without any control option then the program is simply started normally. If an existing process was started with a profile (see profile option), the control option must also provide the same profile name to affect the desired instance.

--nosplash
Start the program without showing the splash screen. This option is useful in conjunction with the --daemon option.
--mode <name>

Start the program with the desired mode. Supported modes are test and dev. These modes are for development purposes and not intended for general use.

The test mode changes the following program behavior:

  • The program is prevented from printing the program header on the command line.
  • The program is prevented from running program checks at startup.

The dev mode changes the following program behavior:

  • The program adds the Development menu on the menu bar.
  • The program adds testing buttons to the Task tool.
  • The program uses target/program as the program home folder.
  • The program home folder is created at startup if it does not exist.
  • The updater folder is changed to target/xenon-updater.
--profile <name>
Start the program with a named profile. Normally this is not necessary but in cases where the user wants to have separate instances of the program running at the same time each instance will need a different profile.
--daemon
Start the program without showing workspaces. This options is useful to start the program "in the background" to allow the program to be running before the user needs to use it. This option can benefit from use of the --nosplash option.
--stop
Stop the program immediately. Using this option does not prompt the user that the program is stopping, will close all workspaces and will attempt to terminate the program normally.
--status
Get the status of the program. If the program is not running then this option prints that the program is not running and exits immediately. If the program is running, this option prints that the program is running and exits this instance of the program immediately, leaving the previously running instance as it was.
--watch
Watch the output of an existing program instance. This instance will continue to watch the existing instance until this instance is terminated, usually by typing Ctrl-C on the keyboard. It is possible to have multiple watching instances for any normal instance.

Logging

Log events are published to stdout and to the program log file by default.

--log-level <level>
The level of logging verbosity. The valid log level values are:

none - Turn off all logging

error - Problematic events that may leave the program in an unstable state

warn - Problematic events that usually leave the program in a stable state

info - Information messages like normal program events (default)

debug - Extra messages that show high level events

trace - Extra messages that show low level events

all - Turn on all logging

--log-file <file>
The log file location. This changes where the log file will be written and, therefore, will not be written in the default location.
--log-append
Should the log file be appended. If this option is not specified the log file will be overwritten each time the program is started.

Appendix A

User Experience Principles

HiDPI Monitors

Support the use of high DPI monitors over the use of standard DPI monitors. Over time the pixel density of monitors had hovered between 80 and 120 DPI. With the introduction of the Apple Retina display in 2010 the DPI increased significantly and now there are many monitors with DPI counts above 150.

Keep the User Context

Whenever the program is restarted it should return the user to where they left off unless the user specifically changes the settings not to. This means that the program should open the same workspaces with the same workareas with the same tools and resources loaded.
